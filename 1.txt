## Network Topology Choices: Ethernet vs InfiniBand / RoCE

### Why Ethernet (400GbE) Instead of InfiniBand

One of the first key design discussions was choosing the **GPU interconnect fabric**. While InfiniBand is often the default for HPC environments, **we deliberately architected a high‑speed Ethernet fabric** for this deployment.

**Decision rationale:**

*   The workloads are a **mix of distributed training, real‑time inference, and Omniverse streaming**, not pure tightly coupled MPI workloads.
*   The customer prioritized **operational simplicity, enterprise networking standards, and multi‑tenant Kubernetes integration**.
*   Advances in **400Gb Ethernet + RDMA (RoCE‑capable design)** allow us to meet performance requirements without operational complexity.

### Final Network Architecture

We designed a **leaf‑spine architecture** optimized for both **east‑west GPU traffic** and **north‑south storage and client access**.

**Key tiers:**

| Fabric Tier                | Purpose                                          | Speed      |
| -------------------------- | ------------------------------------------------ | ---------- |
| **GPU East‑West Fabric**   | Distributed training, NCCL collectives           | **400GbE** |
| **North‑South Fabric**     | Kubernetes, storage access, ingress/egress       | **200GbE** |
| **Storage Backend Fabric** | PowerScale replication and backend data movement | **100GbE** |
| **Frontend / Management**  | ObjectScale, AIDP, backup                        | **25GbE**  |

**Implementation details:**

*   Dual‑spine design using **SN5600 switches** for predictable latency.
*   Non‑blocking fabric ensured consistent performance during multi‑tenant GPU contention.
*   Separate management and data planes to isolate operational traffic from training workloads.

### Challenges Encountered and Resolution

**Challenge:**  
Early validation showed **suboptimal NCCL performance** due to default MTU and congestion control settings.

**Resolution:**

*   Tuned **jumbo frames (MTU \~9000)** across the GPU fabric.
*   Validated **end‑to‑end lossless Ethernet behavior**.
*   Ran **NCCL AllReduce tests across 32 nodes** to establish baseline performance.
*   Validated topology‑aware placement inside Kubernetes using Run:ai.

After tuning, we achieved **stable scaling efficiency across multi‑node GPU jobs**, meeting distributed training targets.

## Storage Fabric and Data Access Design

### Storage Requirements

The platform needed to simultaneously support:

*   **High‑throughput training data access**
*   **Real‑time streaming of digital twins**
*   **Large unstructured asset storage (USD, BIM, CAD files)**
*   Structured analytics and feature pipelines

No single storage system could handle this optimally, so we implemented a **tiered data platform**.

***

### Storage Architecture Overview

**File Storage (Training + Shared Data):**

*   **Dell PowerScale All‑Flash**
*   Exposed to Kubernetes via **CSI driver**
*   Used for:
    *   Model training data
    *   Checkpoints
    *   Shared datasets

**Object Storage (Digital Assets & Omniverse):**

*   **Dell ObjectScale (S3‑compatible)**
*   Used for:
    *   Omniverse USD assets
    *   Digital twin artifacts
    *   Unstructured datasets

**Data Lakehouse Layer:**

*   **Dell AI Data Platform (AIDP)** on top of PowerScale + ObjectScale
*   Supports:
    *   Bronze/Silver/Gold zones
    *   Spark‑based transformations
    *   API‑driven data product access

***

### Storage Fabric Challenges and Mitigations

**Challenge 1: Small‑file metadata pressure from Omniverse**

*   Omniverse workflows generate **many small USD and asset files**.

**Solution:**

*   Placed **Omniverse artifacts on ObjectScale**, which handles metadata fan‑out more efficiently.
*   Used **Omniverse Storage API** as an abstraction layer rather than direct file access.

***

**Challenge 2: Training throughput bottlenecks**

*   Multi‑GPU jobs saturated backend storage during checkpoint writes.

**Solution:**

*   Dedicated **100GbE backend storage fabric** for PowerScale node‑to‑node operations.
*   Tuned Kubernetes CSI parameters and parallel I/O behavior.
*   Validated sustained throughput during checkpoint storms.

*   **CPU‑only Nodes**

    *   Management services
