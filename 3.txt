# **Complex GPU Performance Bottleneck — Debugging a Distributed Training Slowdown**

## **Context**

During a multi‑node distributed training validation in a newly deployed GPU cluster, we encountered a **severe performance regression**:  
NCCL AllReduce jobs that should have scaled linearly across 32 GPUs were hitting **only \~30–40% expected throughput**.

This cluster used:

*   Multi‑GPU PowerEdge systems
*   NVIDIA GPUs
*   Kubernetes + Run:ai scheduler
*   High‑speed Ethernet fabric (400GbE east‑west)
*   NVIDIA GPU Operator, drivers, and NCCL

The slowdown was intermittent, only appearing during multi‑node runs, not single‑node training—which pointed to **interconnect or PCIe/NIC issues**.

***

# **Step 1 — Reproduce & Instrument**

### **Tools Used**

*   `nvidia-smi dmon` for GPU utilization and PCIe throughput
*   NCCL diagnostics:
    ```bash
    NCCL_DEBUG=INFO NCCL_DEBUG_SUBSYS=ALL
    ```
*   `nsys profile` for CUDA kernel and memcpy timing
*   `ethtool -S` for NIC error counters
*   `lldptool` + `lspci -vv` for PCIe link width/speed verification
*   `perf` + `turbostat` for CPU/IRQ profiling
*   NVIDIA DCGM + Run:ai job metrics

### **Initial Findings**

*   GPUs were NOT saturated—even under heavy workloads.
*   NCCL logs showed **spikes in latency** during ring‑exchange phases.
*   NIC counters revealed periodic **RX/TX pause frames**, suggesting congestion or a mis‑tuned RDMA/ECN configuration.
*   One suspicious detail: a subset of nodes reported **PCIe link width degraded from x16 → x8**.

That immediately explained the asymmetric slowdown—an x8 link halves bandwidth, which throttles NCCL collective operations.

***

# **Step 2 — Root Cause Analysis**

### **A. PCIe Link Degradation**

A few nodes had GPUs running at **PCIe Gen4 x8** instead of **x16**, confirmed via:

```bash
lspci -vv | grep -i width
```

Cause:

*   BIOS setting had **“Power Management ASPM”** enabled
*   After reboot, GPU link sometimes negotiated down to x8

### **B. NIC Firmware and Pause Frames**

On affected nodes, Mellanox adapters showed:

*   High **PFC (priority flow control) pause frames**
*   Mismatched **ECN/DSCP settings**
    This caused head‑of‑line blocking for NCCL traffic.

### **C. Kubernetes Scheduling Pattern**

Run:ai placed mixed‑workload containers on the same nodes, occasionally causing:

*   CPU contention on the host
*   Increased IRQ latency, degrading NIC performance

These three issues interleaved to create a significant system‑level bottleneck.

***

# **Step 3 — Remediation Steps**

### **1. Fixing PCIe Throughput**

*   Disabled ASPM and power‑saving PCIe states in BIOS
*   Forced GPU links to train at **x16 Gen4**
*   Verified after reboot across all nodes with automated Ansible checks

### **2. Network & NIC Tuning**

*   Updated NIC firmware and configured a consistent **PFC/ECN profile** across the fabric
*   Enabled **RoCE lane maps** and jumbo frames across the cluster
*   Balanced IRQ affinity using:
    ```bash
    irqbalance --oneshot
    ```
*   Tuned DCQCN congestion control parameters

### **3. Workload & Scheduling Adjustments**

*   Collaborated with the Run:ai team to adjust:
    *   Kubernetes topology hints
    *   Pod anti‑affinity rules
    *   GPU‑exclusive gang scheduling for large NCCL jobs
*   Dedicated GPU nodes during training windows

***

# **Step 4 — Validation**

After fixes, we performed:

*   32‑GPU NCCL AllReduce benchmarks
*   Long‑running mixed workloads
*   Packet‑level telemetry capture to ensure no new pause frames

**Results:**

*   Training throughput improved by **2.3×**
*   Latency spikes disappeared
*   GPU utilization reached expected saturation
*   No further PCIe link‑downs observed

### **B. Non‑Technical Stakeholders (Product owners, leadership)**

I translated findings into business outcomes:

*   “What it enables” → predictable scaling and faster time‑to‑model

Presented a **simple 3‑point summary**:

Resolution: A unified cluster tuning that ensures **maximum GPU throughput and predictable scaling**.
